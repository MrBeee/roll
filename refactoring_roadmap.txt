**Refactoring Roadmap**

The highest-value change is to shrink the orchestration burden in roll_main_window.py:172. Right now the main window is acting as controller, service layer, persistence layer, plot manager, import/export coordinator, and worker-thread owner. That makes almost every change risky. I would refactor in phases so behavior stays stable.

1. Phase 1, extract pure application services without changing UI behavior.
Create a ProjectService for load/save and XML conversion from roll_main_window.py:3031, roll_main_window.py:3673, roll_main_window.py:3735, and roll_main_window.py:3972.
Create a GeometryService for survey rebuild, transform calculation, and validation from roll_main_window.py:3735 and roll_survey.py:560.
Create a FilterService for the duplicated record cleanup handlers in roll_main_window.py:1059.
Create a PlotService for layout and analysis rendering from roll_main_window.py:1951, roll_main_window.py:2292, roll_main_window.py:2342, and roll_main_window.py:2392.

2. Phase 2, replace mutable global runtime state with explicit state objects.
Keep defaults and static constants in config.py, but move mutable runtime values such as surveyCrs, surveyTransform, timers, and debug/session flags out of globals around config.py:256, config.py:257, config.py:258, and config.py:287.
Proposal:
Introduce AppSettings for persisted settings from QSettings.
Introduce SessionState for current open project, recent files, active arrays, and active plots.
Introduce RenderSettings for LOD, colors, and image choices.
This removes hidden coupling between roll_main_window.py, my_parameters.py:1932, and roll_survey.py:399.

3. Phase 3, split UI update work from domain mutations.
Methods like roll_main_window.py:841 are currently doing both domain mutation and screen refresh. Proposal:
One method computes the new survey and returns a result object.
One method applies side effects such as resetting arrays and files.
One method refreshes UI models and plots.
That separation will make failures easier to handle and much easier to test.

4. Phase 4, simplify worker-thread boundaries.
The worker pattern in binning_worker_mixin.py and worker_threads.py is good enough to keep, but I would tighten contracts.
Proposal:
Workers should accept a small immutable snapshot rather than a full XML serialization when possible.
Workers should return only owned output objects rather than writing back into a broad survey object.
Finished handlers should validate shape and metadata once in one place.

5. Phase 5, standardize error handling.
Broad exception use in worker_threads.py:175, worker_threads.py:139, worker_threads.py:175, roll_survey.py:584, and roll_survey.py should be replaced with narrower operational exceptions.
Proposal:
Define a small application error model such as ValidationError, ImportError, GeometryError, and AnalysisError.
Have the UI decide how to present them, but keep error formatting outside the core logic.

6. Phase 6, reduce duplication in import/export and filtering.
The duplicated handlers in roll_main_window.py:1286, roll_main_window.py:1295, roll_main_window.py:1304, roll_main_window.py:1313, roll_main_window.py:1322, roll_main_window.py:1348, roll_main_window.py:1374, and roll_main_window.py:1399 are all good candidates for descriptor-driven helpers.
Proposal:
Use a small table of geometry type, model, getter/setter, export function, and label strings. That will cut repetition without changing behavior.

If I had to prioritize by impact versus cost, I would do:
1. Extract FilterService and ProjectService first.
2. Move runtime globals out of config.py.
3. Split plot logic out of roll_main_window.py:1951.
4. Standardize exceptions.
5. Replace XML round-tripping for internal clones if profiling proves it matters.

**Performance Review**

The core performance risks are not random; they cluster around array churn, duplicate elimination, and UI responsiveness mechanisms.

1. Repeated np.unique calls in geometry generation are likely expensive.
The main ones are in roll_survey.py:663, roll_survey.py:911, and roll_survey.py:1028. Those are expensive because they sort and copy structured arrays, and they appear inside generation flows where data volume is already high.
Proposal:
Generate fewer duplicates up front by changing relation/receiver accumulation strategy.
If duplicates are unavoidable, perform one dedup pass at the end of generation rather than several intermediate passes.
Where only key fields matter, deduplicate on projected keys rather than whole structured rows.

2. Worker completion does a lot of large array copying.
See binning_worker_mixin.py:377, binning_worker_mixin.py:378, binning_worker_mixin.py:379, binning_worker_mixin.py:405, binning_worker_mixin.py:406, binning_worker_mixin.py:407, binning_worker_mixin.py:516, binning_worker_mixin.py:517, and binning_worker_mixin.py:518.
Proposal:
Return immutable result containers and transfer ownership instead of copying when the worker is done.
Only copy arrays that must outlive the worker thread or are backed by temporary buffers.
For memmap-backed outputs, prefer reopening by file path and shape rather than duplicating in memory.

3. Event-loop pumping is being used as a control mechanism.
There are many calls to QApplication.processEvents in roll_main_window.py:3447 through roll_main_window.py:3632, plus repeated calls in my_parameters.py. This can make the UI feel responsive, but it can also create reentrancy problems and unpredictable stalls.
Proposal:
Move long-running import and conversion loops into worker objects with explicit progress and cancellation signals.
Reserve processEvents for narrow cases only, and not inside dense loops.

4. XML is doing double duty as both persistence and in-memory transport.
Deep-copy and worker handoff rely on XML conversion in roll_survey.py:262 and related XML methods around roll_survey.py:3108 and roll_survey.py:3271. That is defensible for correctness, but it is not cheap.
Proposal:
Keep XML for file persistence.
Introduce an internal snapshot object for worker transfer and cloning, containing only fields needed for geometry/binning.

5. Some numerical routines are still explicitly known slow paths.
The comments in functions_numba.py:122 already identify the nested-loop 2D transform as slow. That is a good signal that this is understood but not fully addressed.
Proposal:
Profile actual production-sized inputs first.
If this routine is hot, precompute reusable exponent terms or batch operations in numpy where memory allows.
If memory does not allow vectorization, keep the numba path but isolate parameter grids so repeated runs can reuse them.

6. Rendering is probably acceptable at moderate scales, but the repaint boundary is too coarse.
The progressive painting system in roll_survey.py is a strength, but roll_main_window.py:1951 still appears to rebuild a lot of state. Global replotting after small operations such as duplicate removal is visible in roll_main_window.py:1059.
Proposal:
Invalidate only affected layers instead of calling a full layout replot after every small data mutation.
Keep cached hulls, live/dead partitions, and image layers as separate invalidation domains.

7. Memmap usage is good, but lifecycle handling could be tighter.
Memmap allocation and reopening in binning_worker_mixin.py:182 and binning_worker_mixin.py:411, plus reopening in roll_main_window.py, indicate the right general approach for large analysis data.
Proposal:
Create one small MemmapManager abstraction to own file paths, shapes, modes, flushing, and close semantics.
That will reduce leak risk and make interruption/reload paths easier to reason about.

If you want the shortest list of performance actions with the best likely payoff:
1. Remove repeated unique and copy work.
2. Move import loops off the UI thread.
3. Narrow redraw invalidation after filtering and imports.
4. Replace XML transport for internal snapshots if profiling shows it is material.
