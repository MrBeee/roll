**Refactoring Roadmap**

Updated for the current codebase on 2026-03-27.

The broad direction is still correct: reduce the amount of orchestration living in `roll_main_window.py`, keep domain and persistence logic outside the UI, and make the remaining big refactors safer by growing tests first.

The roadmap below reflects what has already landed and what should move next from the current state, not from the earlier baseline.

**Current Status Summary**

Completed or substantially landed:
1. `ProjectService` now exists and owns project XML read/write plus the low-level `.npy` sidecar helpers.
2. `roll_main_window.py` already delegates most direct sidecar `np.load` and `np.save` calls to `ProjectService`.
3. Analysis sidecars now save at the right moment after full binning, including `.off.npy`, `.azi.npy`, and the refreshed `.ana.npy` timestamp/reopen path.
4. The display and logging dock construction has already been extracted into dedicated builder modules.
5. Worker-thread lifecycle code is already out of `RollMainWindow` and lives in `binning_worker_mixin.py`.
6. The codebase already has a better test baseline than before: project round-trip, sidecar behavior, SPS filters, and recent `functions_numba` regression coverage.

Still unfinished:
1. `fileLoad()` in `roll_main_window.py` is still doing a large amount of orchestration and post-load UI wiring.
2. Duplicate/orphan filtering handlers are still highly repetitive in the main window.
3. Mutable runtime state is still spread across `config.py` and `RollMainWindow`.
4. Import/QC logic and plotting logic are still too close to the UI.
5. Worker contracts still depend on broad survey state and XML snapshots.

**Recommended Order From Here**

1. Extract `FilterService` next.

4. Extract import/QC orchestration from the main window.
5. Extract plot/input preparation from the main window.
6. Clean up worker contracts only after the service boundaries are stable.

The reason for doing `FilterService` before more persistence work is simple: it is a low-risk, high-clarity refactor that removes a large amount of repetition without forcing a redesign of project loading. It also creates another clean service seam, which makes later extraction work easier.

**Phase 0 - Baseline and Safety Net**

Status:
Mostly complete.

What is already in place:
1. Direct tests for project persistence and sidecars.
2. Direct tests for SPS filter logic.
3. Project round-trip coverage.
4. Regression tests for the recent numerical warning fixes.

What is still worth adding here:
1. One or two geometry-generation correctness tests in `roll_survey.py`.
2. One worker success/failure test around `worker_threads.py`.
3. A small reusable project fixture that includes a valid `.roll` plus minimal sidecars.

Exit criteria:
1. Project persistence and filter behavior are protected.
2. At least one geometry-generation path is covered by a direct test.
3. At least one worker path is covered by a direct test.

**Phase 1 - Project Persistence Extraction**

Status:
Partially complete.

What is already done:
1. `ProjectService` handles XML read/write.
2. `ProjectService` handles low-level sidecar save/load helpers.
3. `ProjectService` handles memmap opening helpers.
4. `roll_main_window.py` no longer performs most raw sidecar file I/O directly.

What still remains:
1. `fileLoad()` still contains a long sequence of per-sidecar orchestration, validation, normalization, and UI coupling.
2. Sidecar load policy is still spread between `ProjectService` and `RollMainWindow`.
3. The service does not yet return a structured “loaded project sidecars” result object.

Next sub-step for this phase:
Create one higher-level `ProjectService` method that loads and validates all analysis/data sidecars and returns a structured result object plus warnings. Keep the UI-specific actions in `RollMainWindow`.

Exit criteria:
1. `fileLoad()` becomes much shorter and stops owning sidecar load policy.
2. `ProjectService` becomes the single owner of sidecar naming, existence checks, validation, and load rules.
3. UI code only applies results and refreshes widgets/models.

**Phase 2 - Extract FilterService**

Status:
Not started.

This is now the recommended next step.

Why this should come next:
1. The duplicate/orphan cleanup handlers are repetitive and already built on top of reusable functions in `sps_io_and_qc.py`.
2. The work is relatively low risk because the core algorithms do not need to change.
3. It will remove a large amount of controller noise from `roll_main_window.py`.
4. It is easy to validate with direct unit tests.

Scope:
1. Replace the repeated `removeRpsDuplicates()`, `removeSpsDuplicates()`, `removeXpsDuplicates()`, `removeRecDuplicates()`, `removeSrcDuplicates()`, `removeRelDuplicates()`, and orphan variants with a descriptor-driven service.
2. Keep UI logging, action state changes, and redraw calls in the main window initially.
3. Move filter selection, data replacement, and derived-array refresh rules into `FilterService`.

Implementation approach:
1. Define descriptors for point arrays, relation arrays, and orphan modes.
2. Create generic apply methods for point and relation filters.
3. Return updated arrays and a small summary object containing counts and labels.

Exit criteria:
1. The 14 cleanup/orphan handlers collapse into a small generic dispatch layer.
2. Filter orchestration becomes testable without a `QMainWindow`.
3. `roll_main_window.py` becomes visibly smaller and less repetitive.

**Phase 3 - Explicit Runtime and Session State**

Status:
Not started.

Goal:
Stop using `config.py` and the main window instance as a shared runtime message bus.

What to introduce:
1. `SessionState` for the open project, survey object, imported arrays, geometry arrays, analysis arrays, and project file paths.
2. `AppSettings` for persisted settings backed by `QSettings`.
3. A smaller object for transient profiling/timer state.

What should move out of `config.py`:
1. Mutable timer accumulators.
2. Mutable runtime flags that are not true constants.
3. Any project/session-specific runtime object.

Exit criteria:
1. `config.py` contains defaults, static formats, and style constants only.
2. Services receive state objects rather than many loosely related arrays and flags.

**Phase 4 - Import/QC Service**

Status:
Not started.

Goal:
Move SPS/RPS/XPS parsing and QC orchestration out of the main window.

Scope:
1. File parsing.
2. QC passes.
3. Duplicate/orphan checks that are part of import workflows.
4. Optional progress/cancellation plumbing that can later be moved to a worker.

Important constraint:
Do not move this to a worker first. Extract the logic into a service first, then decide whether a worker is needed.

Exit criteria:
1. Import/QC logic is testable without `RollMainWindow`.
2. Core parsing logic no longer depends on `QApplication.processEvents`.

**Phase 5 - Plot and Redraw Service**

Status:
Not started.

Goal:
Separate domain changes from redraw policy and image preparation.

Scope:
1. Layout image preparation.
2. Analysis image preparation.
3. Colorbar input generation.
4. Partial invalidation rules versus full replot rules.

This phase should happen after `FilterService` and session state are in place, otherwise the plot logic will still be coupled to unstable controller state.

Exit criteria:
1. Data mutations no longer directly decide full redraw policy in many places.
2. Small data edits can invalidate only affected layers.

**Phase 6 - Worker Contract Cleanup**

Status:
Not started.

Goal:
Make worker inputs and outputs explicit, smaller, and easier to validate.

Current note:
The worker setup is acceptable for now. This should not be the next refactor. It depends on earlier service and state extraction to avoid simply moving the same large implicit contract into new classes.

Future direction:
1. Introduce request/result objects.
2. Reduce XML round-tripping for internal worker handoff if profiling shows it matters.
3. Centralize result validation after worker completion.

Exit criteria:
1. Worker inputs and outputs are explicit and documented.
2. Completion handlers do less copying and less implicit mutation.

**Phase 7 - Performance Work After Structure Stabilizes**

Status:
Deferred by design.

The main performance candidates are still:
1. repeated `np.unique` work during geometry generation,
2. large array copying on worker completion,
3. broad redraw invalidation after small changes,
4. XML used as both persistence format and internal transport.

Do not optimize these speculatively. Measure them after the service boundaries and state ownership are clearer.
