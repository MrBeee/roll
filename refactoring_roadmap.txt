**Refactoring Roadmap**

Updated for the current codebase on 2026-04-02.

The broad direction is unchanged: reduce orchestration in `roll_main_window.py`, keep persistence and domain logic out of the UI, and make the remaining large refactors safer by growing tests before moving behavior.

This version reflects the live codebase, not the earlier baseline.

**Current Status Summary**

Completed or substantially landed:
1. `ProjectService` owns project XML read/write, low-level sidecar helpers, memmap opening, and batch project-sidecar loading.
2. `FilterService` exists and the duplicate/orphan handlers in `roll_main_window.py` are now thin dispatch wrappers.
3. Analysis sidecars save at the right moment after binning, including `.off.npy`, `.azi.npy`, and the refreshed `.ana.npy` timestamp/reopen path.
4. Display and logging dock construction has been extracted into dedicated builder modules.
5. Worker-thread lifecycle code is already out of `RollMainWindow` and lives in `binning_worker_mixin.py`.
6. The test baseline is materially better: project round-trip, sidecar behavior, session-service behavior, filter-service coverage, and `functions_numba` regression coverage.

Still unfinished:
1. `fileLoad()` in `roll_main_window.py` still owns too much orchestration and post-load UI/data application.
2. Loaded-sidecar application is still split across `_applyLoadedAnalysisSidecars()` and `_applyLoadedSurveyDataSidecars()` inside the main window.
3. Mutable runtime state is still spread across `config.py` and many `RollMainWindow` instance attributes.
4. SPS/RPS/XPS import and QC logic is still heavily UI-bound.
5. There is still at least one persistence leak outside `ProjectService`: geometry sidecars are written directly from `binning_worker_mixin.py`.
6. Worker contracts still depend on broad survey state and implicit mutation.

**Recommended Order From Here**

1. Finish Phase 1 by extracting loaded-project apply logic out of `fileLoad()`.
2. Remove the remaining direct geometry sidecar writes outside `ProjectService`.
3. Introduce explicit session/runtime state and remove mutable runtime state from `config.py`.
4. Extract import/QC orchestration from the main window.
5. Extract plot/input preparation and redraw policy from the main window.
6. Clean up worker contracts only after service boundaries and state ownership are stable.
7. Do performance work only after the above structure is in place.

The reason to finish the load/apply boundary before deeper state work is simple: the current bottleneck is no longer sidecar loading itself. The service already loads batch sidecars; the remaining complexity is how that loaded state gets applied to UI models, colorbars, derived arrays, and actions in `RollMainWindow`.

**Phase 0 - Baseline and Safety Net**

Status:
Mostly complete.

What is already in place:
1. Direct tests for project persistence and sidecars.
2. Direct tests for SPS filter logic.
3. Direct tests for `FilterService` and `SessionService`.
4. Project round-trip coverage.
5. Regression tests for recent numerical warning fixes.

What is still worth adding here:
1. One or two geometry-generation correctness tests in `roll_survey.py`.
2. One worker success/failure test around `worker_threads.py` or the worker mixin boundary.
3. A small reusable project fixture that includes a valid `.roll` plus minimal sidecars.
4. One higher-level project-load test that exercises sidecar load plus apply behavior.

Exit criteria:
1. Project persistence and filter behavior are protected.
2. At least one geometry-generation path is covered by a direct test.
3. At least one worker path is covered by a direct test.

**Phase 1 - Project Persistence and Load Boundary**

Status:
Mostly complete, but not finished.

What is already done:
1. `ProjectService` handles XML read/write.
2. `ProjectService` handles low-level sidecar save/load helpers.
3. `ProjectService` handles memmap opening helpers.
4. `ProjectService.loadProjectSidecars()` returns a structured batch result with messages and loaded arrays.
5. `roll_main_window.py` no longer performs most raw sidecar `np.load` calls directly.

What still remains:
1. `fileLoad()` still orchestrates parse, reset, batch load, sidecar apply, plotting selection, model setup, and final widget state.
2. `_applyLoadedAnalysisSidecars()` still mixes domain-state application with UI setup such as image selection and colorbar initialization.
3. `_applyLoadedSurveyDataSidecars()` still mixes loaded array assignment, derived-state recomputation, model refresh, and action toggles.
4. `binning_worker_mixin.py` still writes `.rec.npy`, `.rel.npy`, and `.src.npy` directly instead of routing through `ProjectService`.

Next sub-step for this phase:
Create a dedicated loaded-project applier, presenter, or similarly scoped helper that accepts the `ProjectSidecarLoadResult` and applies it to output state, derived arrays, and models. Keep final widget navigation and redraw calls in `RollMainWindow` for now.

Exit criteria:
1. `fileLoad()` becomes much shorter and stops owning most loaded-state application logic.
2. `ProjectService` remains the single owner of sidecar naming, existence checks, validation, and load rules.
3. The remaining main-window code becomes mostly controller wiring: parse, call services, update top-level UI state, and trigger redraw.
4. All sidecar writes go through `ProjectService`.

**Phase 2 - Filter Extraction**

Status:
Complete enough for now.

What has landed:
1. `FilterService` exists and is descriptor-driven.
2. Duplicate/orphan handlers in `roll_main_window.py` now delegate to generic point/relation filter dispatch helpers.
3. The service returns structured filter results including summary text and derived spatial state.
4. There is direct unit-test coverage for point duplicate, point orphan, and relation orphan filtering.

Remaining note:
Do not spend more refactoring effort here unless new duplication appears. The architecture seam is good enough and no longer the next bottleneck.

Exit criteria:
Already met for the current milestone:
1. The 14 cleanup/orphan handlers collapsed into a small generic dispatch layer.
2. Filter orchestration is testable without a `QMainWindow`.
3. `roll_main_window.py` is materially less repetitive in this area.

**Phase 3 - Explicit Runtime and Session State**

Status:
Not started in the intended sense.

Important clarification:
`SessionService` currently handles recent-file menu behavior only. It is not yet a runtime/session-state owner.

Goal:
Stop using `config.py` and the main window instance as a shared runtime message bus.

What to introduce:
1. `SessionState` for the open project, survey object, imported arrays, geometry arrays, analysis arrays, and project file paths.
2. `AppSettings` for persisted settings backed by `QSettings`.
3. A smaller object for transient profiling/timer state.

What should move out of `config.py`:
1. Mutable timer accumulators.
2. Mutable runtime flags that are not true constants.
3. Any project/session-specific runtime object.
4. Session counters such as `surveyNumber`.

Why this should come after finishing Phase 1:
If the current load/apply path stays tangled inside `RollMainWindow`, moving arrays into `SessionState` will mostly relocate the same coupling instead of reducing it.

Exit criteria:
1. `config.py` contains defaults, static formats, and style constants only.
2. Services receive state objects rather than many loosely related arrays and flags.
3. `RollMainWindow` stops owning most imported-array and geometry-array lifecycle directly.

**Phase 4 - Import/QC Service**

Status:
Not started.

Goal:
Move SPS/RPS/XPS parsing and QC orchestration out of the main window.

Current note:
`fileImportSpsData()` is still a large UI-bound routine with repeated `QApplication.processEvents()` calls, direct array allocation, parsing loops, uniqueness checks, orphan checks, CRS conversion, transform calculation, and model refresh.

Scope:
1. File parsing.
2. QC passes.
3. Duplicate/orphan checks that are part of import workflows.
4. Optional progress/cancellation plumbing that can later be moved to a worker.

Important constraint:
Do not move this to a worker first. Extract the logic into a service first, then decide whether a worker is needed.

Exit criteria:
1. Import/QC logic is testable without `RollMainWindow`.
2. Core parsing logic no longer depends directly on `QApplication.processEvents()`.
3. The UI only owns file picking, progress display, and final message/reporting.

**Phase 5 - Plot and Redraw Service**

Status:
Not started.

Goal:
Separate domain changes from redraw policy and image preparation.

Scope:
1. Layout image preparation.
2. Analysis image preparation.
3. Colorbar input generation.
4. Partial invalidation rules versus full replot rules.

This phase should happen after the load/apply extraction and session-state work are in place, otherwise plot logic will still be coupled to unstable controller state.

Exit criteria:
1. Data mutations no longer directly decide full redraw policy in many places.
2. Small data edits can invalidate only affected layers.
3. Colorbar and image setup stop being scattered across controller methods.

**Phase 6 - Worker Contract Cleanup**

Status:
Not started.

Goal:
Make worker inputs and outputs explicit, smaller, and easier to validate.

Current note:
The worker setup is acceptable for now. This should still not be the next refactor. It depends on earlier service and state extraction to avoid simply moving the same large implicit contract into new classes.

Future direction:
1. Introduce request/result objects.
2. Reduce XML round-tripping for internal worker handoff if profiling shows it matters.
3. Centralize result validation after worker completion.

Exit criteria:
1. Worker inputs and outputs are explicit and documented.
2. Completion handlers do less copying and less implicit mutation.

**Phase 7 - Performance Work After Structure Stabilizes**

Status:
Deferred by design.

The main performance candidates are still:
1. repeated `np.unique` work during geometry generation,
2. large array copying on worker completion,
3. broad redraw invalidation after small changes,
4. XML used as both persistence format and internal transport.

Do not optimize these speculatively. Measure them after the service boundaries and state ownership are clearer.
